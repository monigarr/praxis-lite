Component E — Measurement, assertions & gating

Units: U8, U9, U10, U12

The measurement spine: run both arms and compute lift, evaluate each case's assertion, persist the result to a git-committed scoreboard, and expose the CLI + GitLab CI gate that blocks regressions. This is where "the agent gets better" becomes a gated number.

Design decisions

Lift is the unit; cold arm is mandatory (KTD-2). injected arm = CLAUDE.md present + read tool registered; cold arm = neither. Absolute match is never the metric (R1).
Determinism via greedy/low-temp + per-case ε (KTD-5). The gate compares deltas against a per-case epsilon with a conservative global default (R5). No n-runs.
Per-case assertion object (KTD-3). Each case declares its assertion type and parameters; the gate dispatches on it. Phase-1 poison-resistance asserts the poison insight is absent from the generated CLAUDE.md (write-time signal produced in Component B); returns to retrieval-level in Phase 2.
Scoreboard is a git-committed file, updated by the PR (KTD-6). evals/scoreboard/history.jsonl (append one row per green-main run) + evals/scoreboard/latest.json (baseline the gate reads). The PR's diff updates these; CI verifies the committed numbers are real and improve vs latest.json. No external store, no CI write-back.
CI is GitLab CI (KTD-8). The repo is hosted on GitLab (labs.gauntletai.com); the gate is a .gitlab-ci.yml job invoking the harness CLI. (Eval cases may still target GitHub repos — case-target host is independent of CI host.)

U8. Arms orchestration and lift computation

Goal: Run cold and injected arms for a case and compute lift, with greedy/ε determinism settings.
Requirements: R1, R5  ·  Dependencies: U7
Files: evals/arms.py, evals/tests/test_arms.py

Approach: For a case: fetch repo if present (U2), build substrate for injected (U3), run both arms via the injected Runner (U4), grade both (U6), attribute (U7), and produce a CaseResult carrying match_cold, match_injected, lift, attribution, context-control, and run metadata (model_id, substrate). Runs greedy/low-temp; carries the case epsilon. Empty-knowledge cases collapse injected→cold (degenerate lift ≈ 0).

Patterns to follow: pure orchestration over injected collaborators; deterministic under FakeRunner.

Test scenarios:

  • Happy path: injected match > cold match yields positive lift in CaseResult.
  • Degenerate: an empty-CLAUDE.md case yields lift ≈ 0 within ε.
  • Metadata: CaseResult carries model_id, substrate, and per-arm matches.
  • Wiring: repo-bearing case triggers U2 fetch; repo-less case skips it.
  • Covers AE n/a (no origin AEs).

U9. Per-case assertion evaluation

Goal: Evaluate a CaseResult against the case's configured assertion.
Requirements: R2, central-risk guard  ·  Dependencies: U8, U3
Files: evals/assertions.py, evals/tests/test_assertions.py

Approach: Dispatch on assertion.kind:

  • helpfulness: lift > 0 AND no-regression vs baseline (baseline supplied by U10), within ε.
  • poison_resistance (Phase 1): the planted poison insight is absent from the generated CLAUDE.md (write-time credibility/conflict-resolution held); lift may be ≈0 by design. Comment + assertion-param note that this returns to retrieval-level in Phase 2.
  • context_budget: injected context within the case's ceiling (used/present + size from U7).

Returns AssertionResult{passed, reason, evidence}.

Patterns to follow: one dispatch point; each assertion is independently testable.

Test scenarios:

  • Helpfulness pass/fail: positive lift + no regression passes; zero/negative lift fails with a clear reason.
  • Poison resistance pass: poison absent from CLAUDE.md passes even with ≈0 lift.
  • Poison resistance fail: poison present in CLAUDE.md fails with the offending insight named.
  • Context budget: injected size over ceiling fails; under passes.
  • ε boundary: a lift just inside ε of zero is treated as no-lift (assert the boundary).

U10. Scoreboard — git file, verify, compare

Goal: Read/append the git-committed scoreboard and provide the gate's verify + compare operations.
Requirements: R7, R8, KTD-6  ·  Dependencies: U9
Files: evals/scoreboard.py, evals/scoreboard/history.jsonl, evals/scoreboard/latest.json, evals/tests/test_scoreboard.py

Approach: latest.json holds the current baseline per-case scores tagged {kg_version, model_id, substrate, date}; history.jsonl appends one row per green-main run (the compounding curve = its git history). Two gate ops: verify — recomputed CaseResults must match the PR's committed latest.json within each case ε (blocks hand-faked numbers); compare — recomputed scores must pass each case's assertion (U9) against the previous baseline. The harness writes the updated file for the PR author to commit; it never writes back from CI.

Patterns to follow: plain JSON/JSONL, human-diffable; no DB. (Promotion to DynamoDB only if a live dashboard appears — see origin.)

Test scenarios:

  • Append: a green run appends exactly one tagged row to history.jsonl and rewrites latest.json.
  • Verify pass: recomputed scores within ε of committed latest.json pass.
  • Verify fail (fake numbers): committed scores better than recomputed beyond ε fail verification.
  • Compare: recomputed scores passing assertions vs prior baseline pass; a regression fails.
  • Tagging: every row carries {kg_version, model_id, substrate, date}.

U12. CLI and GitLab CI gate

Goal: Expose run and gate CLI entrypoints and wire the gate into GitLab CI.
Requirements: R8, KTD-8  ·  Dependencies: U10, U11
Files: evals/cli.py, .gitlab-ci.yml, evals/tests/test_cli.py

Approach: evals run [case...] runs cases and prints/writes results + updated scoreboard. evals gate takes the diff (via CI-provided changed-file list), selects the slice (U11), reruns it, and performs verify + compare (U10); non-zero exit on any failure blocks the merge. Add a .gitlab-ci.yml job that runs evals gate on changes touching core services, passing the merge-request changed-file set. Document the PRAXIS_GITLAB_TOKEN and model/credentials env the job needs.

Patterns to follow: thin CLI over library calls; exit codes are the gate's contract. GitLab CI rules:changes: to scope the job to core-service paths.

Test scenarios:

  • run happy path: runs a fake-backed case and emits a CaseResult + scoreboard update.
  • gate pass: verify+compare succeed → exit 0.
  • gate fail (regression): a regressed case → non-zero exit with the failing case named.
  • gate fail (fake numbers): verify mismatch → non-zero exit.
  • Test expectation: .gitlab-ci.yml YAML is config — validated by a lint/parse check, not behavior.
