Component A — Case & input layer

Units: U1, U2, U11

The layer that turns on-disk eval cases into runnable inputs and decides, per PR, which cases run. Everything downstream (substrate, runner, grading) consumes the typed EvalCase and the materialized repo checkout produced here.

Design decisions

Repo fetch spans GitHub and GitLab (KTD-7). Code-target cases clone by {url, commit} across both hosts (open-source GitHub + internal GitLab), cached locally by commit; private GitLab uses a token from env.
Diff-scoped runs (KTD-9). A selector maps changed files → the affected case slice (reader change → reader-fixture cases; distiller/generator change → full-loop cases), bounding per-PR cost (R6).

U1. Eval case schema and loader

Goal: Define the EvalCase data model and a YAML loader for the on-disk case format.
Requirements: R2, R6, R7, R10  ·  Dependencies: none
Files: evals/case.py, evals/tests/test_case.py

Approach: A typed model (dataclasses/pydantic — implementer's call from current deps) covering: id, tags, optional repo {url, start_commit} (both-or-neither validation), ingestions[] {type: raw_text|session_log, ref, role: used|ignored}, seed_prompt, target {kind: code|requirements_doc|ticket_list, ...kind-specific}, assertion {kind: helpfulness|poison_resistance|context_budget, params, epsilon?}, grading {deterministic_checks, rubric_id}, entry_point: fixture|raw_ingestion, and a reserved substrate field (defaults claude_md). Loader resolves a case directory (case.yaml + ingestions/ + target/). Validation enforces both-or-neither repo, target-kind/grading coherence, and assertion-kind/params coherence. The schema must admit a bootstrapped origin (R10) without code change — keep provenance fields optional and open.

Patterns to follow: mirror the single-table/dataclass clarity in infra/lib/sessions-table-stack.ts docstrings; keep the schema self-documenting.

Test scenarios:

  • Happy path: a full code-target case YAML loads into the model with all fields populated.
  • Both-or-neither: repo.url without start_commit (and vice versa) raises a validation error.
  • Assertion coherence: a context_budget assertion missing its ceiling param fails validation; a valid one loads.
  • Target coherence: a ticket_list target with code-only grading config fails validation.
  • ε default: a case omitting epsilon loads with the global default applied; an override loads verbatim.
  • Covers AE n/a.

U2. Repo fetch and checkout (GitHub + GitLab)

Goal: Materialize a case's target repo at the specified commit, across GitHub and GitLab, cached by commit.
Requirements: R6 (held fixtures), KTD-7  ·  Dependencies: U1
Files: evals/repofetch.py, evals/tests/test_repofetch.py

Approach: Given {url, start_commit}, clone (or reuse a commit-keyed local cache) and checkout the commit into an isolated working dir. Detect host from URL (github.com vs the GitLab host) only to select auth: public GitHub needs none; private GitLab reads a token from env (e.g., PRAXIS_GITLAB_TOKEN). Shallow-fetch the specific commit where the host allows. Return a context-managed checkout that cleans up. No network calls in unit tests — abstract the git invocation behind a small seam and inject a fake.

Patterns to follow: keep the git/network boundary behind one injectable function so tests stay offline.

Test scenarios:

  • Happy path (GitHub): a public {url, commit} resolves to a checkout dir at that commit (git seam faked).
  • Happy path (GitLab): a GitLab URL selects token auth from env (assert the auth path chosen, git faked).
  • Cache hit: a second fetch of the same commit reuses the cache without re-cloning.
  • Missing token: private GitLab URL with no env token raises a clear, actionable error.
  • Cleanup: the checkout context manager removes its working dir on exit.

U11. Diff-scoped case selector

Goal: Map a PR's changed files to the affected eval slice so CI runs only relevant cases.
Requirements: R6, KTD-9  ·  Dependencies: U1
Files: evals/selector.py, evals/tests/test_selector.py

Approach: Given a changed-file list, map to case tags/entry points: reader changes (evals/knowledge/read_tool.py, future retriever) → reader-fixture cases; substrate/distiller changes (evals/knowledge/claude_md.py) → full-loop cases; grading changes → cases exercising that grader. A configurable mapping table keyed by path globs → case-tag selectors, with a conservative default of "run all" when a change is unmapped.

Patterns to follow: declarative path-glob → tag map; fail safe toward over-inclusion, never silent under-selection.

Test scenarios:

  • Reader change selects only reader-fixture cases.
  • Substrate change selects full-loop cases.
  • Unmapped path falls back to the full suite (assert no silent empty selection).
  • Multiple changed areas union their slices without duplication.
