Component D — Grading

Units: U5, U6, U7

Turns a RunResult into a normalized match score per target type, plus the attribution and context-control metrics that make "the knowledge was actually used, without bloat" measurable. All three target graders compose the same shared primitives (deterministic checks + frozen-rubric judge).

Design decisions

Autonomous grading, deterministic-first (R4). A frozen, versioned rubric + LLM judge where determinism is not possible; credibility/conflict-resolution checks are shared across all target types. The judge LLM client is injected so harness tests stub it.
Context control is first-class, efficiency subsidiary (R3). Attribution tracks present-vs-used knowledge; CLAUDE.md size + used/present ratio are the gating-relevant signals, while tokens/time/turns are recorded but explicitly not part of the gate.

U5. Grading core — deterministic checks, frozen rubric judge, shared checks

Goal: Provide the grading primitives all target types share: deterministic check runner, frozen-rubric LLM judge, and shared credibility/conflict-resolution checks.
Requirements: R4  ·  Dependencies: U4
Files: evals/grading/base.py, evals/grading/rubric.py, evals/grading/shared.py, evals/tests/test_grading_core.py

Approach: Check interface (input: RunResult + target → CheckResult{passed, score, evidence}). RubricJudge loads a frozen, versioned rubric (checklist items, each judged separately with quoted evidence) by rubric_id; the LLM call is injected so tests stub it. shared.py holds credibility and conflict-resolution checks usable by any target type (R4: "shared metrics across target types"). Rubrics live as versioned files under the case target/; humans edit them offline (no runtime human feedback).

Patterns to follow: inject the judge LLM client; keep rubric IDs stable and file-backed for git versioning.

Test scenarios:

  • Happy path: a deterministic check over a known output yields the expected pass/score/evidence.
  • Rubric judge: a stubbed judge over a rubric returns per-item evidence-bearing results; total is the item aggregate.
  • Frozen rubric: loading by rubric_id returns the exact committed checklist; an unknown id errors.
  • Shared check: the conflict-resolution check flags an output that adopts a known-contradictory claim.
  • Determinism: identical inputs to a deterministic check produce identical results across calls.

U6. Per-target graders (code / requirements doc / tickets)

Goal: Implement match scoring for each of the three target kinds, reusing U5 primitives.
Requirements: R4, and origin "three targets ⇒ three graders"  ·  Dependencies: U5
Files: evals/grading/code_target.py, evals/grading/doc_target.py, evals/grading/tickets_target.py, evals/tests/test_target_graders.py

Approach:

  • Code: behavioral-first — run the target repo's tests / case-specified oracle checks against the injected output applied to the U2 checkout; the gold commit/PR is a reference, not a diff answer-key.
  • Requirements doc: required-decision presence (deterministic "must mention X") + frozen-rubric judge for coverage/quality.
  • Tickets: semantic set match — precision/recall of produced units of work vs target units, order-insensitive (judge-assisted equivalence).

Each returns a normalized match score in [0,1] plus evidence, so lift is comparable across kinds.

Patterns to follow: all three compose Check/RubricJudge from U5; no bespoke LLM calls outside the injected judge.

Test scenarios:

  • Code happy path: output passing the oracle checks scores high; output failing them scores low (runner + checks faked).
  • Code non-diff: an alternative-but-correct implementation that passes tests is NOT penalized for differing from the gold commit.
  • Doc: a doc covering all required decisions scores higher than one missing a decision; missing-decision is identified by name.
  • Tickets precision/recall: target of 4 units vs produced 3-correct+1-spurious yields the expected precision/recall.
  • Tickets order-insensitivity: reordering produced tickets does not change the score.
  • Normalization: each grader returns [0,1].

U7. Attribution and context-control metrics

Goal: Capture whether injected knowledge was used (not just present) and the context-control metrics that make "no bloat" measurable.
Requirements: R1 (attribution sub-metric), R3  ·  Dependencies: U6
Files: evals/attribution.py, evals/tests/test_attribution.py

Approach: From a RunResult, classify attribution: caused (output reflects an injected insight — citation/match), correlated, or unused (a win with no injected insight used = flagged false positive for the knowledge system). Context-control metrics for Phase 1: size of injected CLAUDE.md, and used/present ratio (since the whole file is injected, "retrieved more than needed" surfaces as file bloat vs. what the output actually used). Efficiency (tokens/time/turns) is recorded but explicitly subsidiary. A code comment notes that in Phase 2 (selective reader) the present/retrieved distinction becomes real and the poison signal returns to retrieval-level.

Patterns to follow: read the read-tool trace and output evidence captured by U3/U4; no new LLM calls unless used-detection requires the injected judge.

Test scenarios:

  • Caused: output reflecting an injected insight is classed caused.
  • Unused false-positive: a passing injected run that used no injected insight is flagged unused.
  • Context-control: a larger CLAUDE.md with low used/present ratio reports the expected bloat metric.
  • Subsidiarity: efficiency metrics are present but not part of the gating signal returned (assert separation).
