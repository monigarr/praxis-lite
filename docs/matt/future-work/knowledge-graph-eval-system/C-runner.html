Design decisions

Runner behind an adapter (KTD-4). A Runner protocol with a real headless Claude Code adapter (production fidelity, R9) and a deterministic FakeRunner for the harness's own tests so harness logic never depends on live agent calls in CI.
Determinism via greedy/low-temp + per-case ε (KTD-5). Runner configured greedy/low-temp (seed where the tool exposes it); the gate compares deltas against a per-case epsilon with a conservative global default (R5). No n-runs. (The ε comparison itself lives in Component E.)

U4. Runner adapter (real Claude Code + fake)

Goal: Run a seed prompt through an agent under cold or injected conditions, deterministically configurable, behind a Runner protocol.
Requirements: R5, R9, KTD-4, KTD-5  ·  Dependencies: U3
Files: evals/runner/base.py, evals/runner/claude_code.py, evals/runner/fake.py, evals/tests/test_runner.py

Approach: Runner protocol: run(seed_prompt, workdir, knowledge=None, greedy=True) -> RunResult where RunResult carries the produced output (code diff / doc text / ticket list), the tool-call trace (did the read tool fire), and run metadata (model_id, token/time). ClaudeCodeRunner drives real headless Claude Code: places the CLAUDE.md and registers the read tool for the injected arm, neither for cold; configures greedy/low-temp (seed where exposed). FakeRunner returns scripted RunResults for harness tests — no live agent. Live-run wiring is real here but kept thin; broader runner hardening can follow once the real reader lands.

Execution note: Start with the FakeRunner contract test so downstream units (U6–U9) build against a deterministic runner.

Patterns to follow: dependency-inject the runner everywhere; never import the Claude Code adapter into harness-logic modules.

Test scenarios:

  • Happy path (fake): cold run returns its scripted output with no read-tool call; injected run reflects knowledge and records the read-tool call.
  • Greedy config: the adapter passes greedy/low-temp + seed (where available) to the underlying invocation (assert config, live call mocked).
  • Injected wiring: injected mode places CLAUDE.md and registers the read tool in the workdir; cold mode does neither.
  • Metadata: RunResult captures model_id, tokens, and wall-time.
  • Test expectation: real-Claude-Code live invocation is integration-only and mocked in unit tests — reason: no live agent in CI.
