Summary

Build a Python eval harness that measures the lift injected knowledge gives a coding agent — match(injected) − match(cold) — on per-case configured targets (code / requirements doc / ticket list), grades autonomously per run, and exposes a CI gate that blocks PRs to core knowledge services unless they verifiably improve without regression. Phase 1 keeps the knowledge substrate deliberately trivial: the "knowledge graph" is a generated CLAUDE.md file and the "reader" is a Claude tool call that returns the whole file. The harness is built behind thin substrate/reader/runner seams so Phase-2 selective retrieval and richer graphs swap in without reworking measurement, grading, or gating.

This is the measurement spine the proposal calls for (docs/proposal-praxis.md §Spine) — it turns "the agent gets better" from a claim into a gated, replottable number.

Problem frame

Praxis distills session logs into knowledge and injects it into future sessions. The claim — the agent stops relearning the same things — is only credible if measured, and the central risk is negative compounding: one wrong or over-generalized lesson degrades every future session (see origin: Problem & Context). We need an autonomous, versioned eval system that proves injected knowledge (a) helps, (b) resists poisoning, and (c) does so without context bloat, and that can gate changes to the knowledge services in CI.

Both the eval system and the knowledge services it tests are greenfield (repo today holds only session-capture/ and infra/). The plan therefore defines the harness against thin seams and ships a real Phase-1 substrate (CLAUDE.md + whole-file reader tool) rather than blocking on a future KG.

Scope boundaries

In scope (v1): the harness (case schema, fetch, Phase-1 CLAUDE.md substrate + whole-file reader, runner adapter with real Claude Code + fake, deterministic+rubric grading, three target graders, attribution/context-control, lift, per-case assertions incl. poison + budget, git scoreboard with verify/compare, diff selector, CLI + GitLab CI gate, worked example cases).

Deferred for later (from origin)

  • Substrate bake-off as an active dimension (CLAUDE.md vs RAG vs KG vs RAG+rerank). The substrate field exists; v1 is claude_md only and the reader returns the whole file.
  • Bootstrapped case generation from mined sessions (schema admits it; no generator built).
  • Auto-tuned ε from accumulated variance data; v1 uses a conservative default + per-case override.
  • Selective/ranked retrieval reader; arrives in Phase 2 behind the existing reader seam, at which point poison-resistance returns to a retrieval-level assertion.

Deferred to follow-up work (plan-local)

  • Hardening the real headless Claude Code runner (concurrency, sandboxing, flaky-tool handling) beyond the thin Phase-1 adapter.
  • A live dashboard reading the scoreboard (would motivate the DynamoDB promotion noted in origin KTD-6).

Risks & dependencies

  • Single-run noise vs strict gate. Greedy/low-temp + per-case ε mitigate; if flakiness persists, the deferred mitigation is widening ε per noisy case (not n-runs). Risk surfaced explicitly because the gate blocks merges.
  • Real-runner cost/latency in CI. Bounded by the diff-scoped selector (U11) and by unit tests using FakeRunner; only the gated slice runs live.
  • Default ε is a guess (origin open question). Conservative default now; a one-time variance-characterization run later tunes it. Tracked as an assumption.
  • Used-insight attribution signal (origin open question). Phase-1 uses read-tool-call + output evidence; the precise "used" definition is finalized in U7 against what the reader exposes. Decision deliberately co-located with U3/U7 since it shapes the reader contract.
  • Dependency — none external blocks v1: the Phase-1 substrate is real, so the harness does not wait on a future KG. Phase-2 retrieval depends on the reader seam holding (U3).

Open questions (deferred to implementation)

  • Concrete default ε value (per-target-type vs single global) — set during U8/U9, tunable later.
  • Exact "used"-insight signal — finalized in U7 alongside the reader's output trace.
  • Whether the model/credentials for the real runner in CI come from GitLab CI variables or a separate secret store — resolved in U12 wiring.
